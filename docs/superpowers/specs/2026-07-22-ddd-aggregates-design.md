# DDD aggregates → REST — design

**Date:** 2026-07-22
**Status:** Draft (design), doc-only; decisions answered from the best-practice
research (`docs/superpowers/research/2026-07-21-rest-best-practices.md`, item
#12) rather than from maintainer Q&A, per standing instruction.
**Scope:** One (large) increment. Teach the generator a *second* model kind — a
DDD **aggregate root** — alongside the existing anemic `[Model]` (EF) path:
factory-based creation, encapsulated (private-setter / get-only) reads, embedded
value objects, and **command endpoints** for guarded state transitions. No blind
PUT. Builds on the relationship + contract/correctness working tree and reuses
its pagination, ProblemDetails, and TypedResults conventions.

## 0. Why a separate path (research grounding)

The current generator assumes an **anemic** entity: every property has a public
getter *and* setter, creation is `new T { Prop = … }`, and update is a full
field-copy PUT. A DDD aggregate is the opposite — its invariants live behind
private setters, factory methods, and guarded behavior — so the anemic
machinery cannot express it (you cannot object-initialize private setters, and a
blanket PUT would bypass every invariant). The research is explicit
(item #12): the **aggregate root is the only independently addressable
resource**, inner entities/value objects are **embedded** (not independently
addressable), and **guarded transitions become command endpoints**
(`POST /orders/{id}/cancel`), *not* PUT-a-status. This increment implements that
mapping.

## 1. Detecting an aggregate

New attribute (generated by `AttributesGenerator`, alongside `Model`/`Nested`/`Scalar`):

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class AggregateAttribute(string singularName, string pluralName) : Attribute
{
    public string SingularName { get; } = singularName;
    public string PluralName { get; } = pluralName;
}
```

Usage:

```csharp
[EZRestAPI.Aggregate("Order", "Orders")]
public partial class Order
{
    private Order() { }                              // EF materialization ctor

    [EZRestAPI.Factory]
    public static Order Place(CustomerRef customer)  // creation entry point
    {
        return new Order { Customer = customer, Status = OrderStatus.Placed };
    }

    public CustomerRef Customer { get; private set; } = null!;   // read-only outward
    public OrderStatus Status { get; private set; }
    public IReadOnlyList<OrderLine> Lines => lines;              // get-only projection
    private readonly List<OrderLine> lines = new();

    [EZRestAPI.Command]
    public void AddLine(string sku, int quantity) { /* invariant-guarded */ }

    [EZRestAPI.Command("cancel")]
    public void Cancel() { if (Status == OrderStatus.Shipped) throw new InvalidOperationException("Cannot cancel a shipped order."); Status = OrderStatus.Cancelled; }
}
```

Same rules as `[Model]`: `partial`, `int` key (`Id` generated), unique
singular/plural (`EZR002`/`EZR003` apply). A class marked both `[Model]` and
`[Aggregate]` is an error (extend `EZR010`).

## 2. Reads — encapsulation-friendly property collection

Aggregates use a **read-only** property-collection rule distinct from `[Model]`:
include every **public-getter** instance property regardless of setter
accessibility — `{ get; private set; }`, `{ get; init; }`, and **get-only**
`{ get; }` projections all participate (the current `[Model]` rule requires a
setter and so drops get-only members). These feed `Read{Name}Response` and the
paginated `List` exactly as today. Value-object and child-entity properties are
embedded via the existing `[Nested]` owned-type mapping (`{Name}Dto` +
recursive mapper) — no independent routes, per research #12.

`GET /{plural}/{id}` and `GET /{plural}` (paginated envelope from the
relationship increment) are generated unchanged in shape.

## 3. Creation — factory, not object-initializer

An aggregate declares exactly one creation entry point marked
`[EZRestAPI.Factory]`: either a **public static method** returning the aggregate
type, or a **public constructor**.

- `Create{Name}Request` DTO fields = the factory's **parameters** (by name/type);
  value-object parameters nest the VO's `{VO}Dto` and are mapped via the existing
  nested mapper.
- Generated `POST /{plural}` maps the request DTO → factory arguments, invokes
  the factory, `context.{Plural}.Add(result)`, `SaveChangesAsync`, returns
  **201 + Location** (flat canonical URL) + `Create{Name}Response` (research #9).
- **`EZR012` (Error):** an `[Aggregate]` with zero or more-than-one
  `[Factory]` entry point — creation is ambiguous/impossible. Message steers the
  user to mark exactly one static factory or public constructor.

Because creation never object-initializes, private setters are never assigned
from generated code — invariants stay owned by the aggregate.

## 4. Commands — guarded transitions as sub-resources

Public instance methods marked `[EZRestAPI.Command]` (optional name override
`[EZRestAPI.Command("cancel")]`) become **command endpoints**:

- Route: `POST /{plural}/{id}/{command}` where `{command}` is the override or the
  kebab-cased method name (`AddLine` → `add-line`). (Sub-resource form from
  research #11/#12; the `:verb` variant is a later option.)
- Request body: a generated `{Command}{Name}Request` DTO whose fields = the
  method's parameters (VOs nested as DTOs). Parameterless commands take no body.
- Handler: load the **tracked** aggregate (`FirstOrDefaultAsync`, no
  `AsNoTracking`); 404 if missing; map DTO → arguments; invoke the method;
  `SaveChangesAsync`; return **200 + `Read{Name}Response`** (the mutated state,
  research #8/#9 — commands are non-idempotent POSTs that return the new
  representation).
- **Exception → status mapping** (default convention, no user code required):
  - `System.ArgumentException` (and subclasses, e.g. `ArgumentOutOfRangeException`) → **422** ProblemDetails (the command's own argument guard rejected the input).
  - `System.InvalidOperationException` → **409** ProblemDetails (a state/invariant guard refused the transition — "Cannot cancel a shipped order.").
  - Anything else propagates (500). A future `[EZRestAPI]` domain-exception base with an explicit status is the escape hatch.
  - The exception's `Message` becomes the ProblemDetails `detail` (domain messages are written for developers; never contains secrets by DDD convention, but this is documented as a maintainer responsibility).

## 5. Update & delete

- **No general `PUT`** for aggregates — a full-replace would bypass invariants
  (research #8/#12). Free-form attribute editing (a `PATCH` over
  `[EZRestAPI.Editable]` properties) is **out of scope** for v1.
- **`DELETE /{plural}/{id}`** → 204 / 404 (ProblemDetails). Owned value objects
  cascade as today.

## 6. Persistence

- EF Core materializes the aggregate through a **private/protected parameterless
  constructor** (documented requirement; standard DDD-with-EF pattern) and sets
  private-setter properties via its backing-field/reflection support — no public
  setters needed.
- Value objects / child entities map as **owned types** (`OwnsOne`/`OwnsMany`)
  through the existing `DbContextGenerator` nested path.
- The aggregate gets a `DbSet` and participates in `CustomDbContext` exactly like
  a `[Model]`.
- Collection projections exposed as `IReadOnlyList<T>` over a private backing
  field require EF backing-field mapping (`.UsePropertyAccessMode(PropertyAccessMode.Field)`
  where the navigation is field-backed) — generated in `OnModelCreating`.

## 7. Contract conventions (inherited)

All aggregate endpoints use the **contract/correctness** conventions:
`TypedResults` union return types, `application/problem+json` bodies with
`type`/`title`/`status`/`detail`/`code`, `.WithTags("{Plural}")`,
`.WithName(...)` operationIds, and `.ProducesProblem(...)` metadata (404/409/422).
This increment therefore **depends on** the contract/correctness increment
landing first (or re-implements those helpers).

## 8. Architecture (files)

- `AttributesGenerator.cs`: emit `AggregateAttribute`, `FactoryAttribute`, `CommandAttribute`.
- `ProviderExtensions.cs`: `GetAggregates()` provider; an `Aggregate` model record carrying read Properties (new collection rule), the resolved `FactoryInfo` (name, kind, parameters), and `EquatableArray<CommandInfo>` (route name, method name, parameters). Locations stay out of the cached record.
- `DiagnosticsGenerator.cs`: `EZR012` (factory arity), extend `EZR010` (Model+Aggregate).
- `DtoGenerator.cs`: `Create{Name}Request` from factory params; `{Command}{Name}Request` per command; `Read{Name}Response`/list DTOs from read properties.
- `RepositoryGenerator.cs`: `CreateAsync` (factory-invoking), `ReadAsync`/`ListAsync`, `DeleteAsync`, and one `Execute{Command}Async(int id, {Command}{Name}Request request)` per command (tracked load + invoke + save, surfacing the exception outcome).
- `EndpointsGenerator.cs`: POST create, GET read/list, DELETE, and `POST /{plural}/{id}/{command}` per command, with the exception→status mapping.
- `DbContextGenerator.cs`: `DbSet` + owned-type config + field access mode for aggregates.
- `BootstrapGenerator.cs`: register aggregate repositories + map endpoints.
- Example: an `Order` aggregate with a factory, a value object, and `Cancel`/`AddLine` commands.

## 9. Testing

**Fast generator tests (`EZRestAPI.Tests`):**
- Get-only / private-set properties appear in `Read{Name}Response` (and would be dropped under the `[Model]` rule).
- `Create{Name}Request` mirrors factory parameters; POST handler invokes the factory (no object-initializer of the aggregate).
- Each `[Command]` yields a `POST /{plural}/{id}/{command-kebab}` route, a `{Command}{Name}Request`, and an `Execute{Command}Async`.
- Command handler maps `ArgumentException`→422 and `InvalidOperationException`→409.
- No `PUT` route is generated for an aggregate.
- `EZR012` for zero/multiple `[Factory]`; `EZR010` for `[Model]`+`[Aggregate]`.

**Runnable, no-Docker (`Example.Tests`):**
- `/openapi/v1.json` documents the command endpoints and omits PUT for the aggregate.

**Docker integration (`Example.Tests`, "MsSql"):**
- Create an `Order` via `POST /orders`; read it; `POST /orders/{id}/add-line` then read shows the line; `POST /orders/{id}/cancel` succeeds; a second `cancel` after a state that forbids it → 409 problem+json; an invalid `add-line` quantity → 422. Run by the maintainer where Docker exists.

## 10. Out of scope (v1)

`PATCH`/free-form editable properties; multiple factories / named constructors;
command methods with non-void returns other than the aggregate; domain events /
outbox; cross-aggregate references by id (candidate to reuse the relationship
convention later); the `:verb` custom-method URL form; async/long-running
commands (202). Tracked for later cycles.
